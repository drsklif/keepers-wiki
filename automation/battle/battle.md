# Автоматизация боя через Tampermonkey

## Цель

Сделать полностью автоматический бой для всех режимов (арена, гильдия, лига и другие), где боевая механика одинаковая, а отличаются только точки входа в интерфейсе.

Реализация должна работать как клиентский скрипт, загружаемый через Tampermonkey, и обязана выполнять действия в "живом" UI (через клиентские UI-методы и/или взаимодействие с DOM), а не только отправлять команды по сети.

## Что уже известно

- Сервер является источником истины: он валидирует действия, рассчитывает броски/урон и присылает актуальное состояние боя.
- Клиент отвечает за интерфейс, анимации и отправку пользовательских решений.
- Бой состоит из повторяющихся фаз:
  - Фаза 1: выбор карты (`CARD_SELECTION`), сервер присылает список из 3 карт в начале боя и после завершения каждой фазы 2.
  - Фаза 2: выбор/фиксация кубиков (`DICE_SELECTION`), затем завершение фазы и переход к следующему раунду.

## Требуемая логика автопилота

### Фаза 1: выбор карты

- Из 3 доступных карт выбирать "самую мощную".
- Мощность карты брать из серверного конфига/модели игрока, пришедшей в боевом состоянии.
- Важно: это не статический рейтинг карты, а актуальный профиль карты конкретного игрока (уровень, скиллы, камни, звездность и другие улучшения).

### Фаза 2: выбор кубиков

- Фиксировать кубики стихии, соответствующей выбранной карте.
- Дополнительно разрешать фиксацию обычного кубика удара (`HIT`).
- Учитывать ограничение механики: всего 3 броска, но вручную фиксировать можно только на первых 2 бросках; 3-й бросок дает финальный набор.
- После обработки доступных ручных бросков завершать фазу (`BattleDiceRollEnd` через UI-механику клиента).

## Ключевое требование по UI

Скрипт должен работать через интерфейсные действия, например:

- вызов соответствующих UI-методов клиентского кода (предпочтительно);
- при необходимости - поиск и клик по карточкам/кнопкам в DOM;
- ожидание появления нужного состояния экрана;
- контроль таймингов между действиями.

Необходимо избегать "невидимого режима", когда выполняются только прямые сетевые вызовы без UI-взаимодействия.

## Старт и управление автопилотом

- Бой инициирует пользователь вручную.
- Автопилот должен "подхватывать" бой автоматически после входящих событий боя (через сокеты и/или внутреннюю модель событий игры).
- На экране боя должна быть кнопка `Авто`:
  - `Вкл` - автопилот ведет бой;
  - `Выкл` - управление полностью возвращается игроку.
- Переключение должно работать прямо во время боя без перезагрузки страницы.

## Настраиваемые задержки

- После каждого серверного события перед действием автопилота должна быть пауза.
- Пауза нужна для:
  - завершения отрисовки/анимации UI;
  - имитации человеческой реакции игрока.
- Задержка должна быть параметризуемой (например, `reactionDelayMs`, опционально с джиттером).

## Ограничение реализации

- Использовать только клиентский скрипт Tampermonkey как основной инструмент автоматизации.
- Скрипт должен быть переносимым между режимами боя за счет универсального детектора боевого экрана и состояния боя.

## Примерный план решения

1. **Определить архитектуру Tampermonkey-скрипта**
   - Один основной контроллер `BattleAutoPilot`.
   - Подмодули: `StateDetector`, `CardSelector`, `DiceSelector`, `UiActions`, `EventBridge`, `Logger`, `Settings`.

2. **Сделать универсальный вход в бой (независимо от режима)**
   - Определить общие признаки боевого экрана и фаз через DOM/глобальные объекты клиента.
   - Привязать автопилот к состоянию боя и событиям модели, а не к конкретному URL режима.
   - Реализовать "подхват" уже начатого пользователем боя.

3. **Реализовать безопасные UI-действия**
   - Приоритетно использовать штатные UI-методы клиента; fallback - DOM-клик.
   - Для DOM-взаимодействия: проверка видимости/доступности элемента и повторная попытка.
   - Добавить настраиваемую задержку реакции после каждого серверного события.

4. **Реализовать фазу выбора карты**
   - Получить список из 3 доступных карт, присланных сервером.
   - Для каждой карты взять значение мощности из серверной модели профиля игрока.
   - Выбрать карту с максимальной мощностью.
   - Кликнуть карту и подтвердить выбор (если есть отдельная кнопка подтверждения).

5. **Реализовать фазу выбора кубиков**
   - Считать текущие кубики на экране.
   - Фиксировать кубики нужной стихии и `HIT`.
   - Учитывать лимиты: ручная фиксация и reroll доступны только для первых двух бросков.
   - После второго управляемого броска дождаться финального (третьего) и завершить фазу.

6. **Оркестрация цикла боя**
   - Цикл: `CARD_SELECTION -> DICE_SELECTION -> ROUND_END -> ... -> END`.
   - Защита от зависаний: таймауты, проверка изменения состояния, мягкий рестарт логики.

7. **Наблюдаемость и управление**
   - Добавить кнопку `Авто` на экран боя и индикатор текущего статуса.
   - Показать в UI: текущая фаза, выбранная карта, номер броска, состояние `Авто` вкл/выкл.
   - Добавить базовые команды управления: вкл/выкл, пауза (опционально).
   - Логирование решений в консоль для отладки.

8. **Совместимость между точками входа**
   - Протестировать на арене, лиге и гильдийных боях.
   - Вынести режим-специфичные селекторы в конфиг с fallback-правилами.

9. **Надежность и анти-риск**
   - Ограничить частоту действий, добавить ретраи и backoff.
   - Проверить отсутствие конфликтов с ручным управлением пользователем.
   - Добавить "аварийный стоп" при ошибках DOM/состояния.

10. **Итеративное улучшение выбора карты и кубиков**
   - На первом этапе: стратегия "максимальная серверная мощность карты + своя стихия/HIT".
   - На следующем этапе: улучшение эвристики под конкретные карты и контекст боя.

## Критерии готовности (MVP)

- Скрипт Tampermonkey самостоятельно проходит полный бой от старта до завершения.
- Работает минимум в двух разных режимах боя без изменения кода.
- Автопилот корректно подхватывает бой после ручного старта пользователем.
- Все действия визуально отображаются в интерфейсе (через UI-методы клиента и/или клики).
- Есть кнопка `Авто` для мгновенного переключения между авто- и ручным режимом.
- Есть настройка задержки реакции после серверных событий и понятный лог состояния.
