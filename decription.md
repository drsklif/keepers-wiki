# Хранители карт и магии — разбор клиентского проекта

Этот репозиторий — **клиент (front-end) веб-игры**. Большая часть “справочных” данных (карты, предметы, кампания, конфиги баланса) **не зашита в JS-файлах**, а **приходит с сервера по WebSocket** и кладётся в `app.Model.cfg` (см. `js/CFGUtil.min.js`, `keepers.mobi.har`).

## Справочник: карты (структура и характеристики)

Ключевая идея: **карта как сущность описана в конфиге** `app.Model.cfg.cards.list`, а прогресс игрока по карте (уровень/звёзды/комбо) — в `app.Model.user`.

### Поля карты (по `js/CFGUtil.min.js`)

`app.CFGUtil.getCardByID(id)` возвращает объект из `app.Model.cfg.cards.list`. По использованию в клиенте гарантированно/часто встречаются:

| Поле | Смысл |
|---|---|
| `id` | Числовой ID карты. |
| `name` | Имя карты (используется в магазине, интерфейсе). |
| `img` | Путь к картинке карты (например `/images/cards/...`). |
| `quality` | Редкость (0..4), см. таблицу ниже. |
| `default` | Признак “стартовой/дефолтной” карты (для неё мощь в расчёте фиксируется). |
| `et` | Элемент/тип (например `F/N/W/HIT`), влияет на иконку элемента. |
| `combos` | Количество слотов комбо (используется при ценах/прокачке комбо). |

### Редкости (качество) карт

`app.CFGUtil.getCardQualityName(quality)`:

| `quality` | Название |
|---:|---|
| 0 | Простая |
| 1 | Необычная |
| 2 | Редкая |
| 3 | Эпическая |
| 4 | Легендарная |

### Элементы (тип кубиков/магии)

`app.CFGUtil.getElemIMGByType(type)`:

| Type | Картинка |
|---|---|
| `F` | `/images/elem_f_16.png` |
| `N` | `/images/elem_n_16.png` |
| `W` | `/images/elem_w_16.png` |
| `HIT` | `/images/hit_16.png` |

### Мощь карты (как считается)

Расчёт идёт через `app.CFGUtil.getCardPower(cardId, ..., powerMLT, ..., combosLevelsArray)` и использует конфиги:

| Конфиг в `app.Model.cfg` | Роль |
|---|---|
| `cfgCardsPower` | Базовые очки за карту и шаги за уровни комбо (зависит от качества `q0..q4`). |
| `cfgCards.expDeltas` | Требуемый опыт по уровням карты (по качеству). |
| `cfgCards.starsDeltas` | Требования и цена повышения “звёзд” (по качеству). |
| `cfgCards.comboPrices` | Цена улучшения комбо (по качеству и числу комбо). |

Также есть множитель “усиления” (`getCardPowerMLT`): 1 → 1.25, 2 → 1.5, …, 6 → 2.5.

### Что НЕ лежит в проекте статически

В этом проекте **нет полного справочника всех карт** (имя/описание/статы) в виде JSON-файла: он приходит с сервера в `cfg` при авторизации и хранится в `app.Model.cfg.cards.list`.

## Справочник: кампания (миссии) и награды

Клиентский экран кампании — `js/Campaign.min.js`.

### Структура кампании (данные)

Основные данные кампании находятся в `app.Model.cfg.campaign`:

| Путь | Что там |
|---|---|
| `app.Model.cfg.campaign.chapters[]` | Главы кампании. |
| `chapter.id` / `chapter.name` | ID и имя главы (имя видно в UI). |
| `chapter.locations[]` | Список миссий (локаций) в главе. |
| `location.id` | ID миссии/локации. |
| `location.name1` / `location.name2` | Названия (в UI используется `name1` и “короткое” `name2` на пинах). |
| `location.energyCost` | Цена атаки в энергии (используется кнопкой “Напасть за …”). |
| `location.drops[]` | “Возможные награды” за миссию (клиент показывает их в окне миссии). |
| `drop.item` | ID предмета из `app.Model.cfg.items.list`. |
| `drop.chance` | Шанс (используется в расчётах “где найти”/энергозатраты). |

Доступ к этим структурам делается через `js/CFGUtil.min.js`:
- `getChapterByID(id)`
- `getChapterLocationByID(chapter, locationId)`

### Как показываются награды миссии

В `js/Campaign.min.js` в попапе миссии рендерится блок **“Возможные награды”** на основе `location.drops[]`, а предметы берутся через `app.CFGUtil.getItemByID(drop.item)` (img/quality/name).

### Сундуки главы (награды за звёзды)

В кампании есть 3 сундука на главу, привязанные к числу звёзд:

| Сундук | Требование |
|---:|---|
| 1 | 10 звёзд |
| 2 | 20 звёзд |
| 3 | 30 звёзд |

Открытие идёт командой `cmd: 'CampaignChestOpen'`, а сама начинка сундука приходит с сервера как `drop` и обрабатывается через `UserUtil` (`processChestDrop`).

### Что лежит статически в `Campaign.min.js`

В файле зашита **только “геометрия” карты**: фон (`/images/maps/mapX.jpg`), координаты 10 миссий на карте главы и координаты сундуков. Список глав в этом UI ограничен `1..13`.

## Справочник: предметы и крафт (структура)

В `js/CFGUtil.min.js` есть “справочные” функции по предметам, которые опираются на:
- `app.Model.cfg.items.list` (список предметов)
- `item.craft` (рецепт, если предмет крафтится)
- `item.price` (если продаётся напрямую)

Полезные функции (как ориентиры в коде):

| Функция | Что делает |
|---|---|
| `getItemByID(id)` | Достаёт предмет. |
| `getItemPriceByID(id)` | Рекурсивно считает цену крафта (если крафтится) или возвращает `price`. |
| `getItemSellPriceByID(id)` | Цена продажи (конвертация в золото, деление/округление). |
| `getWhereToFindItemByID(id)` | Ищет по `campaign.chapters[].locations[].drops[]`, где падает предмет. |
| `getItemEnergyCost(id)` | Оценивает “энерго-стоимость” предмета по шансам дропа и `energyCost` миссий. |

## Другая справочная информация (похожая на справочник)

Ниже — куски, которые в коде явно описаны “таблично” и часто нужны при разборе механик.

### Колесо фортуны (Roulette)

Используются конфиги `app.Model.cfg.cfgRoulette`:

| Поле | Смысл |
|---|---|
| `cfgRoulette.list[]` | Сегменты/параметры рулетки (множитель `mlt`, лимиты и т.п.). |
| `cfgRoulette.attempt.goldRate` | Курс конвертации цены попытки в золото. |
| `cfgRoulette.attempt.prices[]` | Цена попыток (структура `{t, v}`), выбирается по номеру попытки. |

### Квесты: человекочитаемые названия

`app.CFGUtil.getQuestName(quest)` маппит `quest.type` → отображаемую строку. Там есть как ежедневные (`d_*`), так и сюжетные/прогрессовые (`lc`, `cc`, `cs`, `lu`, руны/сферы и т.д.).

### Достижения: названия и описания

`app.CFGUtil.getAchievementName(code)` и `getAchievementInfo(achievement)` задают **справочник текстов** для достижений.

### Подземелья: расписание открытия

`app.CFGUtil.getDungeonNextOpenTime(type, now)` считает ближайшее открытие по типу (`HIT/F/N/W`) и дням недели.

## Что полезного в `keepers.mobi.har`

Файл `keepers.mobi.har` содержит трассировку загрузки и **лог WebSocket-сообщений** (включая payload’ы).

### Транспорт и протокол

| Наблюдение | Где видно |
|---|---|
| Используется `socket.io` | В запросах на `js/libs/socket.io.v2.2.0.js`. |
| Идёт WebSocket-сессия | В HAR как `"_resourceType": "websocket"` и массив `"_webSocketMessages"`. |
| Команды от клиента | Сообщения вида `42["command", {"cmd":"Auth","num":1}]` и далее `cmd: 'WorldInfo'`, `PVPTBattle`, `BattleUpdate` и т.п. |
| События от сервера | Сообщения вида `42["events","{...}"]` с `type: "UPDATE"`, `CHAT_MESSAGE_NEW` и др. |

### Конфиги, приходящие с сервера

В HAR видно, что сервер при авторизации присылает большой объект:
- `server.time`
- `cfg: {...}` (например, `cfgRoulette` точно присутствует)

Это подтверждает, что **справочники карт/кампании/предметов/баланса живут в `cfg`**, а клиент — в основном UI + вычисления по этому `cfg`.

### Примеры “структуры боевых данных”

В WebSocket-пакетах в HAR встречаются объекты боя, где у карт есть:
- `card` (ID карты),
- `level`,
- `stars`,
- `combos[]` (например `hit`, `strength`, `type`),
что помогает понять, какие поля реально гоняются между клиентом и сервером.
