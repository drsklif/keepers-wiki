# Анализ боя (arena) по `battle.har`

## Что проверял

- Лог WebSocket в `battle.har` (клиентские команды `Battle*` и ответы сервера с объектом `battle`).
- Клиентский бой в `js/BattlePage.min.js`.
- Расчёт параметров карт/комбо в `js/UserUtil.min.js`.

## Короткий ответ: что считается где

### На сервере

- Валидация всех действий боя (`BattleCardSelected`, `BattleDiceRoll`, `BattleDiceRollEnd`).
- Генерация результатов броска кубиков (фактические значения типов кубов в ответе `battle.m1.dices`).
- Расчёт урона/лечения/эффектов раунда.
- Пересчёт HP обеих сторон.
- Смена состояния боя (`CARD_SELECTION`, `DICE_SELECTION`, `ROUND_END`, `END`).
- Определение исхода (`winner`), включая ничью.
- Формирование очередного набора карт на выбор в фазе выбора карты (сервер присылает актуальный `battle`).

### На клиенте

- Отрисовка интерфейса, анимаций, таймеров, подсветок.
- Отправка команд с выбором игрока:
  - `BattleCardSelected` (индекс карты),
  - `BattleDiceRoll` (массив фиксации кубов),
  - `BattleDiceRollEnd`.
- Локальный предпросмотр/описания урона карты (статы комбо) для UI.
- Визуальное обновление HP-полос, когда сервер уже прислал новые значения.

## Подтверждение из `battle.har`

Последовательность в логе:

1. `BattleUpdate` -> сервер возвращает `state=CARD_SELECTION`.
2. `BattleCardSelected` -> сервер возвращает `state=DICE_SELECTION`.
3. Несколько `BattleDiceRoll` -> сервер каждый раз возвращает новый `battle` с `dices`.
4. `BattleDiceRollEnd` -> сервер возвращает `state=ROUND_END` и уже **новые HP**.
5. Снова `BattleUpdate` -> сервер переводит в `CARD_SELECTION` следующего цикла.

В этом логе таких циклов 3, затем:

- после последнего `BattleDiceRollEnd`: `m1hp=0`, `m2hp=0`,
- `BattleUpdate` -> `state=END`, `winner=0` (ничья).

Это прямое подтверждение, что итог раунда/боя вычисляет сервер, а клиент только отображает присланное состояние.

## Важный нюанс по кубикам

В командах клиента:

- `BattleDiceRoll` отправляет только массив `dices` (какие кубы зафиксированы).

В ответе сервера:

- приходит полный набор кубов с типами (`F/N/W/HIT`) и флагом фиксации (`s`).

То есть клиент **не генерирует “рандом кубов” окончательно**, он передаёт только выбор фиксации; результат броска приходит с сервера.

## Что видно по коду `BattlePage.min.js`

- Бой инициализируется через `a.command({cmd:"BattleUpdate"}, ...)`.
- На выбор карты: `a.command({cmd:"BattleCardSelected", idx: ...}, ...)`.
- На конец фазы кубов: `a.command({cmd:"BattleDiceRollEnd"}, ...)`.
- После ответов клиент подменяет локальную модель `h=e.battle`, `u.m1.model=h.m1`, `u.m2.model=h.m2` и рисует UI.

Это паттерн authoritative server: сервер владеет истинным battle-state.

## Про боевую/защитную колоды

В клиенте явно видно разделение:

- Обычная боевая колода: `app.Model.user.cards.deck` (используется в обычных боях/арене).
- Отдельная колода защиты для гильд. войн: в `UserUtil` есть `isCardInGWDefenceDeck`, проверка через `app.Model.user.gw...d.l`.

В `battle.har` анализируемый бой — arena (`t=2`), т.е. используется обычная боевая колода, не GW-defence.

## Выводы

1. Сервер полностью authoritative для механики боя: состояние, броски, урон, HP, исход.
2. Клиент не “считает исход”, а отправляет решения игрока и визуализирует ответы сервера.
3. Клиентские расчёты статов карты важны для UI (что карта может сделать), но финальный результат раунда определяется сервером.
4. Модель с 2 повторяющимися фазами, которую вы описали, полностью подтверждается логом (`CARD_SELECTION` <-> `DICE_SELECTION`, затем `ROUND_END`).
